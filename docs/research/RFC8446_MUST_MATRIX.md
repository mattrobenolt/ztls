# RFC 8446 MUST coverage matrix

This matrix is the working claims-to-evidence map for TLS 1.3 normative
requirements. It is deliberately conservative: a requirement is not marked as
proved unless there is named test evidence, clear structural evidence in a
single code path, or an explicit out-of-scope decision tied to a GitHub issue.

Status terms:

| Status | Meaning |
|---|---|
| `PROVEN` | Covered by named tests, external conformance evidence, or a small auditable implementation invariant. |
| `PARTIAL` | Some evidence exists, but either the test is indirect or the requirement is only partly enforced. |
| `GAP` | In current supported surface but missing implementation, missing direct evidence, or both. |
| `OUT-OF-SCOPE` | Not in the current ztls supported surface; tracked by the issue named in the row. |
| `CALLER` | Required behavior belongs to the Sans-I/O caller/application boundary, not the core TLS engine. |
| `N/A` | Requirement is for a role ztls does not implement, a legacy TLS mode ztls forbids, or middlebox behavior. |

The current supported surface is the one named in `PRODUCTION_READINESS.md`:
TLS 1.3 full handshake over X25519, the three TLS 1.3 cipher suites ztls
implements, certificate-authenticated server flight with explicit client
verification gates, application data, alerts, `close_notify`, post-handshake
KeyUpdate, and parse/discard handling for NewSessionTicket. HRR, PSK/resumption,
0-RTT, client authentication, P-256/P-384/PQ groups, exporters, and TLS 1.2
fallback are not supported today.

Rows below group tightly related RFC sentences only when they have the same
implementation path, evidence, and disposition. A grouped row is still a
normative claim: splitting it would duplicate the same evidence and status. The
issue asks for MUST/MUST NOT/SHALL plus security-relevant SHOULDs; SHOULD-level
rows are kept in the same matrix but the row text names them as `should` rather
than promoting them to MUSTs.

## Current high-value gaps exposed by the matrix

These are not the only gaps above, but they are the ones most likely to matter
for the next correctness slice:

- ServerHello downgrade-sentinel check is not explicitly implemented/tested.
- ClientHello parser/server path needs direct tests for missing
  `supported_versions` and mixed unknown/known cipher-suite lists.
- CCS compatibility handling should validate the payload byte and allowed window.
- `close_notify` should transition the engine into an ignore/reject-after-close
  state or the caller-boundary contract should explicitly say this is outside
  the core engine.
- Immediate zeroization of intermediate secrets and old traffic keys after
  KeyUpdate is not proven.
- MTI requirements not met or not proven: RSA-PSS certificate signatures,
  P-256 key exchange, and live cookie/HRR support.
