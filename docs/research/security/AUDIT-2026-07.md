## Methodology

1. Six scoped agents ran in parallel against fixed attack-class/scope pairs
   (client state machine, server state machine, certificate path, FFI crypto
   seam, record layer/framing, mechanical class-bug sweep). Raw reports:
   `.tmp/audit-2026-07/hunt-*.md` (scratch, not committed).
2. The parent then read every production file in `src/` directly and
   **independently verified or rejected every agent claim**. Where agents
   disagreed (the mechanical sweep missed two live panic sites the hunters

3. Verification labels:
   - **CONFIRMED-REPRO** ‚Äî reproduced with a runnable test/PoC (parent-run).
   - **CONFIRMED-SOURCE** ‚Äî verified by direct code reading; the bug is

## Confirmed vulnerabilities

### S5 ‚Äî CRITICAL: chain validation accepts an end-entity certificate as an issuing CA (server-authentication bypass)

**Verification:** CONFIRMED-REPRO (parent-ran PoC; ztls accepts a chain OpenSSL rejects).

**Root cause:** `certificate.parse` / `parseClientChain` verify each
certificate's signature with the next certificate in the presented chain
(`src/certificate.zig:199-200,284`) and anchor the last one in the trust
bundle (`src/certificate_policy.zig:146-156`), but **nowhere is
`basicConstraints.cA == TRUE` or `KeyUsage.keyCertSign` enforced for any
certificate used as an issuer**. Leaf policy (`verifyLeafAuth`) is applied
only to `cert_index == 0`. `certificate_parser.zig` does not even parse
BasicConstraints.

**Exploit chain:** an attacker holds *any* ordinary end-entity certificate
(`CA:FALSE`, `KeyUsage: digitalSignature`) issued by a root the client
trusts ‚Äî e.g. their own cheap domain cert. They mint a forged leaf for
`victim.com` signed with their end-entity key and send
`[forged victim.com leaf, their end-entity cert]` as the Certificate chain.
ztls verifies the forged leaf's hostname and leaf policy, verifies its
signature with the attacker's end-entity key, verifies the end-entity against
the trusted root, and returns the forged leaf public key. CertificateVerify
then proves possession of the forged key. **Any ztls client with a trust
bundle accepts an impersonated server.** The same primitive applies to mTLS
client-chain validation server-side.

**PoC:** `.tmp/audit-2026-07/poc-cert/end_entity_ca_bypass_test.zig` (+ OpenSSL-
generated root/attacker/victim fixtures). `openssl verify` rejects the chain
(`invalid CA certificate`, `key usage does not include certificate signing`);
`ztls.certificate.parse` with a real bundle and `host_name` accepts it.

**Fix direction:** parse BasicConstraints and KeyUsage on every certificate;
require `cA == TRUE` (and `keyCertSign` when KeyUsage is present) for any
certificate consumed as an issuer (chain positions ‚â• 1 and the trust-anchor
lookup); honor `pathLenConstraint`. Negative test = the PoC fixture set.

### S6 ‚Äî HIGH: pre-auth remote DoS ‚Äî u24 narrow-arithmetic overflow in server ClientHello reassembly (#72 class, site missed by two sweeps)

**Verification:** CONFIRMED-REPRO (parent PoC panics with `integer overflow`).

**Root cause:** `src/ServerHandshake.zig:1467`:
```zig
const body_len = (@as(u24, body[1]) << 16) | (@as(u24, body[2]) << 8) | body[3];
const total = handshake_header_len + body_len;   // u24 arithmetic
```
Zig 0.15 evaluates `comptime_int + u24` in `u24`; `body_len = 0xFFFFFF`
overflows before `total > self.ch_buf.buffer.len` can reject. Panic in
Debug/ReleaseSafe, UB in ReleaseFast. The split-header reassembly variant has
the identical bug at `src/ServerHandshake.zig:1423`
(`self.ch_expected = handshake_header_len + body_len_assembled;`).

**Input shape:** a single 9-byte plaintext record
`16 03 03 00 04  01 FF FF FF` to `ServerHandshake.handleRecord()` in
`wait_ch`. No keys, PSK, or caller misbehavior needed ‚Äî any ztls server
process reachable over the network.

**PoC:** `.tmp/audit-2026-07/poc-record/ch_overflow_verify.zig` (parent-written).

**Fix direction:** widen before adding:
`const total = @as(usize, handshake_header_len) + @as(usize, body_len);`
at both sites. Regression test asserts `IncompleteRecord`, not panic. Note the
mechanical sweep (`sweep-mechanical.md`) verified the 16 fixed sites but
missed this one ‚Äî its grep pattern didn't model `comptime_int + narrow_var`.

### S7 ‚Äî HIGH: pre-auth remote DoS ‚Äî u8 narrow-arithmetic overflow in ClientHello `psk_key_exchange_modes` parse (#72 class, third live site)

**Verification:** CONFIRMED-REPRO (parent PoC panics with `integer overflow`). Found
by the parent's own read-through after both agent sweeps missed it.

**Root cause:** `src/client_hello.zig:930`:
```zig
const list_len = ext[0];              // u8, attacker-controlled
if (list_len + 1 != ext.len) return error.InvalidExtensionLength;
```
`list_len + 1` evaluates in `u8`; `list_len = 0xFF` overflows before the
comparison. Panic in safe builds, UB in ReleaseFast.

**Input shape:** any ClientHello containing a `psk_key_exchange_modes`
extension with `ext_len = 2`, first byte `0xFF`. Pre-auth, no PSK config
needed ‚Äî the parser runs before any policy.

**PoC:** `.tmp/audit-2026-07/poc-record/psk_kem_overflow_verify.zig`
(parent-written).

**Fix direction:** `if (@as(usize, list_len) + 1 != ext.len)`. Add the
regression test. **Process note:** this class has now produced 19 confirmed
sites across three audits; the `ziglint` gap called out in AGENTS.md is real
‚Äî a dedicated lint rule or fuzz-target list for `narrow + comptime_int`
should be treated as unfinished business from #72.

### S8 ‚Äî HIGH: public AEAD API accepts mismatched in/out slice lengths ‚Üí libcrypto out-of-bounds write

**Verification:** CONFIRMED-REPRO (agent PoC: 31 adjacent guard bytes overwritten by
`EVP_EncryptUpdate`; parent verified the code path).

**Root cause:** `src/aead.zig:89,110` ‚Äî `Aead.encrypt`/`Aead.decrypt` check
only suite equality, never `ciphertext.len == plaintext.len`. The OpenSSL
backend (`src/crypto/backend_openssl.zig:438,466`) passes `plaintext.len` to
C while writing to `ciphertext.ptr`; C has no slice length. AWS-LC/BoringSSL
wrappers `assert` equality (trap in safe builds, optimizer-assumption UB in
ReleaseFast) instead of returning an error.

**Reachability:** not peer-reachable today ‚Äî `RecordLayer` decrypts in place
with equal slices (parent-verified). Reachable from the public `ztls.aead`
API and a direct hazard for the C ABI (#30) where slice-pair discipline is
per-language. A caller that mis-slices corrupts adjacent memory.

**PoC:** `.tmp/audit-2026-07/poc-ffi/aead_mismatch_overwrite.zig` (`guard
bytes changed: 31`).

**Fix direction:** validate length equality (and `ad`/nonce/tag pointers) in
the public `Aead.encrypt`/`decrypt` before the backend call; return a typed
error. Keep the backend asserts as defense-in-depth.

## Protocol / state-machine findings (CONFIRMED-SOURCE)

### S9 ‚Äî server: ClientHello2 is not validated against ClientHello1 after HRR (RFC 8446 ¬ß4.1.2 MUST)

`src/ServerHandshake.zig:593-814` ‚Äî `processClientHelloMessage` parses CH2 and
unconditionally overwrites `selected_alpn` and `client_server_name`; only the
key_share group and suite are pinned to the HRR. Nothing compares CH2's SNI,
ALPN, cipher_suites, signature_algorithms, psk_key_exchange_modes, or PSK
offer against CH1, per the RFC's "the server MUST check that the ClientHello
differs only in the permitted ways" requirement. The public
`clientServerName()` accessor is documented as available "after the first
handleRecord/acceptClientHello call returns" ‚Äî which includes the HRR return ‚Äî
so a caller that routes/selects credentials on CH1 SNI or ALPN can be handed a
CH2 with different values and complete the handshake under the mutated
identity. Exploitability depends on caller behavior (tenant routing, cert
selection, auth policy), but ztls currently makes the unsafe caller pattern
the natural one. **Fix:** snapshot CH1's security-relevant fields at HRR time
and reject CH2 deltas outside ¬ß4.1.2's permitted set with `illegal_parameter`;
document that SNI/ALPN are unstable until the ServerHello write event.

### S10 ‚Äî server: PSK selection ignores `psk_key_exchange_modes` and PSK/cipher-suite hash compatibility (two RFC MUSTs)

`src/ServerHandshake.zig:351-407` (`selectPsk`) never reads
`parsed.psk_key_exchange_modes`, so a client offering only `psk_ke` (or
omitting the extension) with a valid binder is resumed in `psk_dhe_ke` mode ‚Äî
violating RFC 8446 ¬ß4.2.9/¬ß4.2.11 ("MUST also select a key establishment mode
from the set indicated by the client"). Separately, selection doesn't check
that the negotiated suite's hash matches the ticket's hash (¬ß4.2.11.2: "the
server MUST ensure that it selects a compatible PSK and cipher suite"); an
AES-256/SHA-384 ticket can be selected while negotiating AES-128/SHA-256, with
the early secret derived under the negotiated hash. Not an auth bypass (the
binder still gates on PSK knowledge; Finished binds the transcript), but a
protocol-policy bypass that breaks strict peers and 0-RTT parameter
consistency. Agent PoC sketch: `.tmp/audit-2026-07/poc-server/psk_mode_bypass.zig`.

### S11 ‚Äî client: EncryptedExtensions `early_data` acceptance is not cross-checked

`src/ClientHandshake.zig:1393` copies `ee.early_data_accepted` into
`server_accepted_early_data` with no check that (a) the server actually
selected the PSK (`server_selected_psk`), (b) no HRR intervened, or (c)
`early_tx` was actually installed. Two compounding shapes:
- `offered_extensions` is captured from CH1 and never refreshed after HRR, so
  the EE gate still admits `early_data` post-HRR even though 0-RTT cannot
  survive HRR (early_tx was deinit'd).
- `startWithPsk(offer_early_data=true)` emits the `early_data` extension even
  when `ticket.max_early_data_size == null` (i.e. offers early data it cannot
  send); `early_tx` is only installed when the ticket permits it.
`clientFinished` then silently skips EndOfEarlyData when `early_tx == null`
while leaving `server_accepted_early_data == true`. The live transcript/Finished
mismatch makes these fail closed (handshake dies) rather than exploitable, but
the client accepts protocol states RFC 8446 ¬ß4.2.10/¬ß4.5 forbids and reports a
lying `server_accepted_early_data` flag to the caller. **Fix:** reject EE
`early_data` unless `server_selected_psk && early_tx != null &&
retry_selected_group == null`; only encode CH `early_data` when the ticket
permits it.

### S12 ‚Äî client: trailing handshake bytes after server Finished in the same flight are silently dropped

`src/ClientHandshake.zig:1321-1329,1346-1349` ‚Äî `processFlightBytes` /
`processFlightBuffer` return immediately when state reaches `.send_finished`
without draining the reader (and `processFlightBuffer` clears the buffered
remainder). A flight `EE||Cert||CV||Finished||<garbage handshake message>`
verifies and connects, silently discarding bytes that are illegal at that
point. The server side explicitly rejects the symmetric case
(`processClientFinishedPlaintext`: `if (try hr.next() != null) return
error.UnexpectedMessage`). Not an auth bypass (Finished must already verify),
but a fail-open asymmetry that hides peer misbehavior from the transcript
boundary. **Fix:** require the reader to be drained when `.send_finished` is
reached.

### S13 ‚Äî client: HRR path returns an unframed ClientHello2 as a `.write` event

`src/ClientHandshake.zig:1019-1024` + `1167-1185` ‚Äî `start()` wraps the
ClientHello in a record (`out[frame.header_len..]` + header), but
`processHelloRetryRequest` returns raw `client_hello.encodeRetryAfterHrr`
output (handshake bytes, first byte `0x01`), and `handleRecord` surfaces it as
`.write`. Callers write `.write` bytes to the transport verbatim, so an HRR
makes the client emit malformed wire bytes (record type 0x01) and the
connection dies. Reachable today only via `injectClientHello`/custom CH (the
default start path offers key shares for every advertised group, so legal HRR
is normally impossible) ‚Äî but the 0-RTT fuzz setup and the low-level API hit
it directly. **Fix:** frame CH2 exactly like `start()` does, and bind the
existing HRR test to assert the returned bytes parse as a handshake record.

### S14 ‚Äî mTLS: no client-certificate identity is surfaced (design gap, security-relevant)

`ServerHandshake` verifies the client chain and CertificateVerify, retains the
leaf *public key* ‚Äî and then discards everything. No subject/SAN/identity is
exposed to the caller, so a deployment doing mTLS gets "a key we verified"
with no bound identity to authorize against. Combined with S5 (issuer CA
policy not enforced) this is weaker than it looks: today the only binding is
signature-verified chain to a trust anchor plus leaf EKU/KU. Not a code bug,
but an audit-visible hole in the authentication story.

## Hardening findings (CONFIRMED-SOURCE unless noted)

**H1 ‚Äî Non-constant-time PSK binder comparison.** `src/ServerHandshake.zig:423`
uses `std.mem.eql` on the offered binder; `finished.zig:44` already uses
`crypto.timing_safe.eql` for Finished. Trivial fix, consistent hygiene.

**H2 ‚Äî EC private scalars freed with `BN_free`, not `BN_clear_free`.**
`src/crypto/backend_openssl.zig:247,319` ‚Äî the temporary BIGNUM holding the
P-256/P-384 private scalar is freed without clearing limbs; also reachable via
`signature.PrivateKey.fromP256Scalar`. Secret residue in libcrypto heap.

**H3 ‚Äî `aeadInit` (OpenSSL) doesn't validate key length.**
`src/crypto/backend_openssl.zig:409-427` passes `key_bytes.ptr` to
`EVP_EncryptInit_ex` with the length implied by the cipher; a short slice from
any future caller is an OOB read. Unreachable today (typed `Aead` keys are
fixed-size) ‚Äî enforce in the backend or at the facade.

**H4 ‚Äî ML-KEM wrapper trusts provider-returned lengths.**
`src/mlkem.zig` / `backend_openssl.zig` KEM paths return provider-reported
lengths without pinning to X25519MLKEM768 constants (pub 1216 / ct 1120 /
secret 64). A provider bug or ABI drift becomes a variant DHE input instead of
an error.

**H5 ‚Äî KEM group echo confusion (latent).** `ServerHandshake` KEM selection
encapsulates with hardcoded `"X25519MLKEM768"` but echoes the *offered* hybrid
group in ServerHello (`k.group`). A client offering `secp256r1_mlkem768` /
`secp384r1_mlkem1024` fails closed today (length mismatch ‚Üí ECDHE fallback),
but if OpenSSL ever accepts the bytes, SH would claim one group while the
ciphertext is another's. Pin `k.group == .x25519_mlkem768` before attempting
encapsulation.

**H6 ‚Äî X.509: unknown critical extensions are ignored** (HYPOTHESIS‚Üíagent PoC
passed). `src/certificate_parser.zig:1271-1273` `continue`s on unrecognized
OID before checking criticality; RFC 5280 ¬ß4.2 requires rejection of
unrecognized critical extensions.

**H7 ‚Äî X.509: SAN GeneralName class confusion** (HYPOTHESIS‚Üíagent PoC passed).
`certificate_parser.zig:640-649` switches on the tag *number* only; a
universal-class tag 2 (INTEGER) with content `"victim.com"` matches as
dNSName. Requires a CA to sign malformed DER.

**H8 ‚Äî X.509: EKU extension accepts a non-SEQUENCE wrapper**
(HYPOTHESIS‚Üíagent PoC passed). `certificate_parser.zig:538-550` never checks
the extnValue content is a SEQUENCE before iterating it as OIDs.

**H9 ‚Äî X.509: UTCTime century bug + lax time grammar** (HYPOTHESIS‚Üíagent PoC
passed). `certificate_parser.zig:1353-1367` maps every YY to 20YY; RFC 5280
requires 00‚Äì49‚Üí20YY, 50‚Äì99‚Üí19YY. A cert with UTCTime `notAfter` YY‚â•50 (long
expired) reads as valid for decades. GeneralizedTime is length-checked only
(fractional seconds / missing `Z` tolerated), and `Date.toSeconds` accepts
impossible dates (Feb 31).

**H10 ‚Äî X.509: duplicate extensions are last-wins, not rejected**
(HYPOTHESIS). The extension loop overwrites stored slices; RFC 5280 forbids
duplicate extensions in a certificate. TLS CertificateEntry extensions are
deduped correctly ‚Äî this is the X.509 layer.

**H11 ‚Äî client PSK/suite hash compatibility not enforced.**
`ClientHandshake.offered_psk_cipher_suite` is stored (line 485,738) and never
read; a server selecting the PSK under a different-hash suite is accepted.
Server-side MUST per ¬ß4.2.11.2 (see S10); client-side defensive check is
missing.

**H12 ‚Äî `post_handshake_count` counts NewSessionTickets on the client.**
`ClientHandshake.receiveConnected` increments the flood counter for every
message (NST included) ‚Üí 17 tickets in one record (or 17 ticket records
without interleaved app data) trips `TooManyKeyUpdates`. Meanwhile the
reset-on-any-app-data rule makes the KeyUpdate-flood bound toothless against
an actual flood (peer interleaves 1-byte app records). Server side counts only
KeyUpdates. Align semantics: count KeyUpdates only, and don't reset on app
data.

**H13 ‚Äî `receiveApplicationData` bypasses the KeyUpdate-fragment interlock.**
`src/ServerHandshake.zig:1790` decrypts without the `ku_frag.len != 0` check
that `handleConnected` enforces ‚Äî app data interleaved inside a fragmented
KeyUpdate is accepted if a caller mixes the low-level helper with
`handleRecord`. CONFIRMED-REPRO (agent PoC).

**H14 ‚Äî 0-RTT byte counter saturates at `maxInt(u32)`.**
`src/ServerHandshake.zig:1519-1522` uses `+|=` then `>`; with
`max_early_data_size == 0xffffffff` the counter pins at the limit and never
trips past it. Requires a valid resumed PSK + that exact policy value + >4 GiB
of early data; low impact, exact arithmetic. Replace with a
check-before-adding form.

**H15 ‚Äî C ABI: KeyUpdate/NST events are silently swallowed.**
`capi.zig` maps both to `.none` while the engine marks `pending_write`; the C
consumer gets no bytes, no signal, and the next call wedges on
`err_pending_write`. Documented-partial, but the failure mode is silent ‚Äî
either surface the events or reject KeyUpdate at the C boundary. Related red
flag: `ztls_client_init` hardcodes `insecure_no_chain_anchor = true` (no
server authentication at all) until #30's certificate slice lands ‚Äî the C ABI
remains an *unauthenticated* TLS client by construction until then.

**H16 ‚Äî `Signer` is a borrowed context with no lifetime guard.**
`signature.zig:40-47` ‚Äî `Signer.context` aliases `PrivateKey`; `deinit()` then
use = UAF into libcrypto. Documented, but the C ABI will make this easier to
get wrong. Consider a credential object that owns the lifetime.

**H17 ‚Äî Systemic secret residue in HKDF temporaries.** `hkdf.zig` passes
`Prk`/`TrafficSecret` by value through `deriveSecret`/`expandLabel` chains;
the canonical copies in `suite_state`/`RecordLayer` are securely zeroized but
stack temporaries (early/master secrets, traffic secrets) are not. The
handshake paths wipe the important locals (`makeHandshakeArm`,
`installHandshakeKeys` do); the by-value ABI copies remain. Compiler and
call-site dependent ‚Äî worth a dedicated zeroization strategy pass.

**H18 ‚Äî `sendEarlyData` doesn't mark `pending_write`.**
`src/ClientHandshake.zig:929-934` ‚Äî caller can interleave an unacknowledged
early-data record with other engine writes; engine-level desync footgun, not
peer-reachable.

**H19 ‚Äî `certificate.parse`/`parseClientChain` skip the handshake body-length
check** (unlike `parseClientCertificate`). Safe in the live path
(`HandshakeReader` slices exactly), inconsistent for direct public callers.

**H20 ‚Äî 0.5-RTT interop bug (not security):** a server's application_data
arriving while the client is in `.send_finished` (legal post-Finished 0.5-RTT
data) hits `error.UnexpectedRecord` because the state‚Üíconnected transition
happens only after the caller writes the client Finished.

**H21 ‚Äî hostname/policy gaps worth a decision (not bugs per ztls's current
scope):** CN fallback when SAN absent (RFC 6125 deprecated), wildcard
`*.com`-style public-suffix matches accepted (CA/B BRs forbid issuance;
browsers reject), SHA-1 accepted in chain signatures (interop tradeoff,
discouraged by RFC 8446 ¬ß4.4.2.2), no IDNA/A-label handling. Each is a policy
call ‚Äî flag for explicit acceptance or fix.

**H22 ‚Äî `entropy.fillLinux` panics on unexpected `getrandom` errno.**
Fail-stop is arguably right for a CSPRNG, but it's a panic in library code;
decide whether an error return is more honest for embedders.

**H23 ‚Äî `rejectDuplicateExtensions` is O(n¬≤) over attacker extension blocks.**
Bounded by message size (~8k min-size extensions in a 32 KB block ‚Üí ~33 M
comparisons) per pre-auth connection attempt. Cheap amplifier, no memory
impact; consider a sort+scan or early size cap.

**H24 ‚Äî `ServerHandshake.client_cert_policy` field default is
`.leaf_usage = .server_auth`** ‚Äî `init` overrides it to `.client_auth`, but
the struct-literal default is a trap for direct-construction callers (client
certs would be checked for the *server* EKU). Make the field required or fix
the default.
